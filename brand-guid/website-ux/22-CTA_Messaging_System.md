# CTA Messaging System (AECS)

## 1. Core Principle
CTAs are not marketing prompts. They are engineering decision triggers that move the user from understanding → system engagement → project definition.

---

## 2. CTA Hierarchy

### Level 1: System Exploration CTA
Purpose: Move user into understanding AECS.

- Explore System Architecture
- Understand Engineering Systems
- View Ceiling Systems Logic

---

### Level 2: Trust Building CTA
Purpose: Reduce uncertainty.

- View Engineering Proof
- Explore Case Studies
- See Execution Projects

---

### Level 3: System Engagement CTA
Purpose: Engage user with technical depth.

- Analyze System Design
- Review Engineering Logic
- Compare System Types

---

### Level 4: Conversion CTA (Critical)
Purpose: Convert user into RFQ.

- Request Engineering Analysis
- Submit Project for Evaluation
- Define Your Project as a System

---

## 3. CTA Language Rules

### Allowed
- Engineering language
- System-based verbs
- Analytical tone

### Not Allowed
- Buy now
- Contact us (as primary CTA)
- Marketing persuasion
- Emotional urgency

---

---

## 5. Psychological Conversion Model

CTAs follow this flow:

Awareness → Understanding → Trust → Validation → Action

---

## 6. Engineering Principle

A CTA is successful when:

- It reduces ambiguity
- It increases system clarity
- It leads to structured input (RFQ)

---

## 7. Final Rule

Every CTA must behave like a system command, not a sales trigger.
